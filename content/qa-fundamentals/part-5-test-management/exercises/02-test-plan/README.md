# Exercise 02 — Test Plan

**Technique focus:** Test Planning

**Time:** 40 min

## Mission

You have reviewed the requirements. Now you write the plan. A test plan tells everyone — developers, product managers, the release manager — exactly how the QA team will test the TaskForge 2.0 release.

Your test plan must cover the full release: scope, objectives, test types, entry and exit criteria, environment, risks, and deliverables. Write it in `starter/test-plan.md`.

## What makes a good test plan

A test plan is not just a checklist. It is a contract between QA and the rest of the team. A good plan:

- Defines what is in scope and what is not (no surprises later)
- States measurable exit criteria so "done" is objective
- Identifies risks before they happen so there are mitigations ready
- Lists what the team will deliver at the end of testing

## Deliverable

A completed `starter/test-plan.md` with all 8 sections filled in.

## How to verify

Open `test-plan.md` and check:

- All 8 sections (H2 headings) are present and not empty
- Scope describes what is in and out of scope for this release
- Exit criteria are measurable (contain numbers or percentages)
- Risks and Mitigations lists at least 2 risks with corresponding mitigations

## Pass condition

- Sections `Scope`, `Test Objectives`, `Entry Criteria`, `Exit Criteria`, and `Risks and Mitigations` all pass the grader's word count and keyword checks.
- Each passing section has at least 15–20 words of meaningful content.
- Sections use vocabulary appropriate to the testing domain.
